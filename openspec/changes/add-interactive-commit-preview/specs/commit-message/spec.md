## 新增需求

### 需求：生成语义化提交信息
系统必须使用 LLM 生成符合 Semantic Git Commit Message 标准的提交信息。

#### 场景：存在暂存变更时生成提交信息
- **当** 用户在有暂存变更的仓库中运行 CLI
- **那么** 系统应输出符合语义化格式的提交信息

### 需求：仅支持 OpenAI 兼容的 Chat Completion API
系统必须只通过 OpenAI 兼容的 Chat Completion API 完成模型调用。

#### 场景：执行模型调用
- **当** 系统向模型发送生成请求
- **那么** 必须使用 OpenAI 兼容的 Chat Completion API 进行调用

### 需求：全英文 CLI 界面与帮助
系统的命令行交互与帮助信息必须为英文。

#### 场景：用户触发帮助或错误
- **当** 用户查看帮助或遇到错误
- **那么** 系统应以英文提供说明与提示

### 需求：开箱即用与最小化配置
系统必须在最小化配置下可用，并提供合理默认值。

#### 场景：用户未提供额外配置
- **当** 用户以默认参数运行
- **那么** 系统应能够生成提交信息而不要求额外配置

### 需求：融入现有 git 工作流
系统必须与现有 git 工作流无缝协作，不替代 git。

#### 场景：用户执行提交流程
- **当** 用户使用该工具生成提交信息
- **那么** 系统应以不打断既有 git 流程的方式提供结果

### 需求：性能优先的生成策略
系统必须以降低模型推理开销为优先目标，避免长时间等待。

#### 场景：生成提交信息
- **当** 系统调用模型生成提交信息
- **那么** 应采用以性能为优先的生成策略

### 需求：良好的 CLI 用户体验
系统必须提供清晰的 loading 状态与建设性的错误提示。

#### 场景：模型调用失败
- **当** 模型请求失败
- **那么** 系统应输出建设性的错误提示并保持英文界面

### 需求：模块化与目录结构约束
系统必须将代码放在 src/ 目录下，并将界面逻辑、LLM 逻辑、Git 信息获取逻辑分离。

#### 场景：代码组织
- **当** 实现相关模块
- **那么** 系统应在 src/ 下按职责分离模块并保持高内聚低耦合
